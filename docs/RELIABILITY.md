# TaskFlow — 可靠性策略

## 可用性目标

- 99.5% 正常运行时间（团队协作工具可接受水平）
- 计划维护：每周日 02:00–04:00 UTC

## 错误处理策略

### 后端
- 带错误码的结构化错误响应
- 部分失败时事务回滚
- 通知投递失败使用死信队列
- 优雅降级：数据看板统计可接受短暂过期，任务操作不可

### BFF
- 含指数退避的重试（最多 3 次）
- 熔断器：后端 30 秒内失败 5 次，则 60 秒内返回 503
- 超时：单次后端调用 10 秒，聚合调用 30 秒
- 降级：后端不可用时返回缓存数据（仅限读端点）

### 前端
- TanStack Query 自动重试失败请求（3 次尝试）
- 任务拖拽使用乐观更新，失败时回滚
- 错误边界捕获渲染失败
- Toast 通知展示用户可感知的错误

## 数据库可靠性

- 通过 Prisma 连接池（最多 10 连接）
- 数据看板查询使用读副本（未来）
- 自动每日备份
- 每个迁移配有回滚脚本

## Redis 可靠性

- 用途：限流、会话缓存、热点任务数据
- Redis 不可用时优雅降级（回退到数据库）
- 最大内存策略：allkeys-lru

## 健康检查

- `/api/v1/health` — 后端存活（DB + Redis ping）
- `/api/bff/health` — BFF 存活（后端 API 可达性）
- 所有容器配置 Docker 健康检查

## 可观测性

- 结构化 JSON 日志（pino）
- 按端点的请求耗时直方图
- 按领域的错误率追踪
- 告警：5 分钟内错误率 > 5%
