# TaskFlow - 团队任务协作平台 技术方案

## 1. 项目概述

**项目名称**：TaskFlow  
**项目定位**：面向小型团队的轻量级任务协作平台，支持看板管理、任务分配、进度追踪与团队协作  
**核心价值**：提供直观的看板视图与实时协作能力，让团队高效管理任务流转

### 1.1 核心功能

| 模块 | 功能 |
|------|------|
| 用户系统 | 注册/登录、JWT鉴权、个人设置 |
| 工作区 | 创建/加入工作区、成员管理、角色权限(Owner/Admin/Member) |
| 看板 | 创建看板、自定义列(Todo/InProgress/Review/Done)、拖拽排序 |
| 任务 | 创建/编辑/删除任务、指派成员、标签、优先级、截止日期、子任务 |
| 评论 | 任务内评论、@提及成员 |
| 通知 | 任务状态变更通知、@提及通知、截止日期提醒 |
| 数据看板 | 任务完成统计、成员工作量分布、燃尽图 |

## 2. 技术架构

### 2.1 整体架构

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Frontend   │────▶│     BFF      │────▶│    Backend    │
│   Next.js    │     │    Hono      │     │   Next.js     │
│  (App Router)│     │  (Standalone)│     │ (API Server)  │
└──────────────┘     └──────────────┘     └──────────────┘
      │                     │                     │
   UI渲染/交互          聚合/裁剪/鉴权         业务逻辑/数据持久化
   React Components     请求编排/转换           Domain Service
   Tailwind + shadcn    错误统一处理           Prisma ORM
```

### 2.2 三层职责划分

| 层级 | 技术 | 职责 | 不做 |
|------|------|------|------|
| **Frontend** | Next.js App Router | UI渲染、用户交互、页面路由、客户端状态管理 | 直接调用Backend API |
| **BFF** | Hono (独立部署) | 请求聚合与裁剪、JWT验证与角色校验、数据格式转换(适配前端)、错误统一包装、速率限制 | 不含业务逻辑、不直接操作DB |
| **Backend** | Next.js (standalone模式) | 核心业务逻辑、数据校验、数据库CRUD、文件上传处理、事务管理 | 不关心前端展示格式 |

### 2.3 请求流转示例

```
前端页面 → BFF: GET /api/bff/boards/:boardId/tasks
          → BFF聚合调用:
            1. Backend: GET /api/v1/boards/:boardId/tasks (任务列表)
            2. Backend: GET /api/v1/boards/:boardId/members (成员列表)
          → BFF合并数据，将memberName注入task，裁剪无用字段
          → 返回前端精简结构
```

## 4. 技术选型

### 4.1 前端 (apps/web)

| 类别 | 技术 | 说明 |
|------|------|------|
| 框架 | Next.js 15 (App Router) | RSC + CSR混合渲染 |
| UI库 | shadcn/ui + Radix | 可定制、无样式锁定 |
| 样式 | TailwindCSS 4 | 原子化CSS |
| 状态管理 | Zustand | 轻量，支持SSR hydration |
| 数据请求 | TanStack Query v5 | 缓存、乐观更新、自动重试 |
| 拖拽 | @dnd-kit/core | 看板列拖拽排序 |
| 表单 | React Hook Form + Zod | 类型安全的表单校验 |

### 4.2 BFF层 (apps/bff)

| 类别 | 技术 | 说明 |
|------|------|------|
| 框架 | Hono v4 | 轻量、多运行时、TypeScript优先 |
| 运行时 | Node.js (hono/node-server) | 独立部署 |
| 校验 | Zod | 请求/响应类型安全校验 |
| HTTP客户端 | ky | 基于fetch，支持重试与拦截器 |
| 鉴权 | @hono/jwt + 自定义中间件 | JWT验证+角色RBAC |

### 4.3 后端 (apps/api)

| 类别 | 技术 | 说明 |
|------|------|------|
| 框架 | Next.js 15 (standalone) | API Routes + Server Actions |
| ORM | Prisma 6 | 类型安全DB访问 |
| 数据库 | PostgreSQL 16 | 主数据存储 |
| 缓存 | Redis 7 | 会话、速率限制、热点数据 |
| 校验 | Zod | 请求体校验 |
| 鉴权 | jose (JWT) | Edge-compatible JWT库 |
| 文件存储 | S3兼容 (MinIO开发) | 头像/附件上传 |
| 密码 | bcryptjs | 密码哈希 |

### 4.4 测试

| 类别 | 技术 | 覆盖范围 |
|------|------|----------|
| 单元测试 | Vitest | Service函数、Validator、DTO转换、工具函数 |

## 5. 数据库设计

### 5.1 ER关系

```
User ──< WorkspaceMember >── Workspace ──< Board ──< BoardColumn ──< Task
  │                                                   │
  └──< TaskAssignee >─────────────────────────────────┘
                                        │
  └──< Comment >────────────────────────┘
Task ──< SubTask
Task ──< TaskLabel >── Label
Task ──< Notification
```

### 5.2 核心表结构

**User**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID (PK) | 主键 |
| email | VARCHAR(255) UNIQUE | 登录邮箱 |
| name | VARCHAR(100) | 显示名称 |
| passwordHash | VARCHAR(255) | bcrypt哈希 |
| avatarUrl | VARCHAR(500) | 头像 |
| createdAt | TIMESTAMP | 创建时间 |

**Workspace**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID (PK) | 主键 |
| name | VARCHAR(100) | 工作区名称 |
| description | TEXT | 描述 |
| createdAt | TIMESTAMP | 创建时间 |

**WorkspaceMember**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID (PK) | 主键 |
| workspaceId | UUID (FK) | 关联工作区 |
| userId | UUID (FK) | 关联用户 |
| role | ENUM | OWNER/ADMIN/MEMBER |

**Board**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID (PK) | 主键 |
| workspaceId | UUID (FK) | 关联工作区 |
| title | VARCHAR(200) | 看板标题 |
| description | TEXT | 描述 |

**BoardColumn**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID (PK) | 主键 |
| boardId | UUID (FK) | 关联看板 |
| title | VARCHAR(100) | 列标题 |
| position | INT | 排序位置 |

**Task**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID (PK) | 主键 |
| columnId | UUID (FK) | 所在列 |
| title | VARCHAR(300) | 任务标题 |
| description | TEXT | 详细描述 |
| priority | ENUM | LOW/MEDIUM/HIGH/URGENT |
| position | INT | 列内排序 |
| dueDate | TIMESTAMP | 截止日期 |

**TaskAssignee / Comment / Label / SubTask / Notification** — 略，按常规设计

## 6. API设计

### 6.1 Backend API (内部，BFF调用)

Base URL: `http://api:3001/api/v1`

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /auth/register | 注册 |
| POST | /auth/login | 登录(返回JWT) |
| GET | /workspaces | 工作区列表 |
| POST | /workspaces | 创建工作区 |
| GET | /workspaces/:id/members | 成员列表 |
| POST | /workspaces/:id/members | 邀请成员 |
| GET | /boards?workspaceId= | 看板列表 |
| POST | /boards | 创建看板 |
| PATCH | /boards/:id | 更新看板 |
| GET | /boards/:id/columns | 获取看板列 |
| PATCH | /boards/:id/columns/reorder | 列排序 |
| GET | /tasks?columnId= | 任务列表 |
| POST | /tasks | 创建任务 |
| PATCH | /tasks/:id | 更新任务 |
| PATCH | /tasks/:id/move | 移动任务到其他列 |
| DELETE | /tasks/:id | 删除任务 |
| POST | /tasks/:id/comments | 添加评论 |
| GET | /notifications | 通知列表 |
| PATCH | /notifications/:id/read | 标记已读 |

### 6.2 BFF API (前端调用)

Base URL: `http://bff:3002/api/bff`

| 方法 | 路径 | 说明 | 聚合逻辑 |
|------|------|------|----------|
| POST | /auth/register | 注册 | 透传，返回时剥离passwordHash |
| POST | /auth/login | 登录 | 透传，返回时附加用户基本信息 |
| GET | /workspaces/:id/overview | 工作区概览 | 聚合: 工作区信息+成员列表+看板列表+任务统计 |
| GET | /boards/:id/detail | 看板详情 | 聚合: 看板信息+列(含任务)+成员映射表，一次返回完整看板渲染数据 |
| POST | /boards | 创建看板 | 透传+创建默认列 |
| PATCH | /tasks/:id/move | 移动任务 | 透传移动+触发通知创建(调通知API) |
| GET | /me/notifications | 我的通知 | 聚合: 通知列表+关联任务标题+触发人信息 |

## 7. 测试策略

### 7.1 测试分层

```
                    ┌─────────────┐
                    │  E2E Tests  │  ← Playwright (少量，验证核心流程)
                    │   5-10个    │
                    ├─────────────┤
                    │Integration  │  ← Vitest + MSW + testcontainers
                    │   Tests     │     (中等数量，验证API与BFF聚合)
                    │  30-50个    │
                    ├─────────────┤
                    │ Unit Tests  │  ← Vitest (大量，验证逻辑单元)
                    │  100+个     │
                    └─────────────┘
```

### 7.2 单元测试 (Vitest)

**覆盖目标**：`apps/api/src/services/` + `apps/api/src/validators/` + `apps/bff/src/dto/` + `packages/shared/`

| 模块 | 测试内容 | 示例 |
|------|----------|------|
| Backend Service | 业务逻辑函数 | `TaskService.create()` — 校验→创建→返回；`TaskService.move()` — 校验列存在→更新position→触发通知 |
| Validator | Zod Schema校验 | 合法输入通过、缺失字段拒绝、类型错误拒绝、边界值校验 |
| BFF DTO | 数据转换函数 | `transformBoardDetail()` — Backend数据→BFF响应格式，验证字段裁剪与嵌套重组 |
| Shared Utils | 纯函数 | `formatDueDate()`、`calculateBurndown()` |

**运行方式**：`vitest run --project api --project bff --project shared`

### 7.3 集成测试 (Vitest + MSW + testcontainers)

**Backend集成测试**：

```
真实PostgreSQL容器(testcontainers) → Prisma → Repository → API Route
```

- 启动PG容器，执行migration
- 通过`next-test`或直接调用API Route handler
- 验证完整请求-响应链路，包括DB状态变更

| 场景 | 验证点 |
|------|--------|
| POST /auth/register | 创建用户、密码哈希、重复邮箱拒绝 |
| POST /auth/login | 正确密码返回JWT、错误密码返回401 |
| POST /boards | 创建看板、自动创建默认4列、非成员403 |
| PATCH /tasks/:id/move | 移动任务、更新position、跨列移动、列不存在404 |

**BFF集成测试**：

```
MSW拦截Backend API → BFF Route → 聚合逻辑 → 响应
```

- 使用MSW模拟Backend响应
- 验证BFF的聚合、裁剪、错误处理逻辑

| 场景 | 验证点 |
|------|--------|
| GET /boards/:id/detail | 两次Backend调用被正确发起、数据正确合并、无关字段被裁剪 |
| GET /workspaces/:id/overview | 多次Backend调用聚合、统计字段计算正确 |
| Backend返回500 | BFF统一包装为前端友好错误格式 |
| JWT过期 | BFF返回401，不透传Backend错误 |

### 7.4 E2E测试 (Playwright)

**测试用户流程**：

| 编号 | 流程 | 步骤 |
|------|------|------|
| E2E-1 | 完整注册到任务完成 | 注册→创建工作区→创建看板→添加任务→指派成员→拖拽到Done列→验证通知 |
| E2E-2 | 协作流程 | 用户A创建任务并@用户B→用户B登录查看通知→用户B评论→用户A看到评论 |
| E2E-3 | 权限控制 | Member尝试邀请成员→失败→Admin邀请成功 |
| E2E-4 | 看板定制 | 添加自定义列→重命名列→删除列→拖拽列排序 |
| E2E-5 | 数据看板 | 查看任务统计→验证燃尽图数据→切换时间范围 |

**运行方式**：`playwright test`，CI中使用`playwright test --workers=2`

## 8. CI/CD流水线

### 8.1 CI流程 (PR触发)

```yaml
# .github/workflows/ci.yml
触发条件: pull_request (main分支)

jobs:
  lint-and-typecheck:
    runs-on: ubuntu-latest
    steps:
      - Checkout
      - Setup pnpm + Node 20
      - Install dependencies (pnpm install --frozen-lockfile)
      - Turbo run lint typecheck       # ESLint + tsc --noEmit

  unit-test:
    runs-on: ubuntu-latest
    needs: lint-and-typecheck
    steps:
      - Turbo run test:unit             # Vitest --coverage
      - Upload coverage to Codecov

  integration-test:
    runs-on: ubuntu-latest
    needs: lint-and-typecheck
    steps:
      - Start PostgreSQL (Docker service)
      - Run migration
      - Turbo run test:integration      # Vitest

  e2e-test:
    runs-on: ubuntu-latest
    needs: [unit-test, integration-test]
    steps:
      - Build all apps
      - Start services (docker-compose)
      - Install Playwright browsers
      - Turbo run test:e2e
      - Upload Playwright report (on failure)

  build-verify:
    runs-on: ubuntu-latest
    needs: [unit-test, integration-test]
    steps:
      - Turbo run build                 # 确保所有app可构建
```

## 10. 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| BFF独立部署 vs Next API Route集成 | **独立部署** | 职责隔离，BFF可独立扩缩容，不污染Frontend构建 |
| Next.js Backend vs Express/Fastify | **Next.js** | 统一技术栈降低学习成本，API Route够用，standalone模式部署轻量 |
| Prisma vs Drizzle | **Prisma** | 迁移系统成熟、类型生成稳定、社区资源丰富 |
| Zustand vs Jotai | **Zustand** | 看板拖拽场景需要全局store，Zustand更直观 |
| testcontainers vs 内存DB | **testcontainers** | 真实PG行为一致，避免SQLite/PG差异导致的测试盲区 |
| MSW vs Nock | **MSW** | BFF集成测试需模拟Backend，MSW支持Service Worker模式，更贴近真实 |
