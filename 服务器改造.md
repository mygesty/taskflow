# 台式机改造为 Jenkins + 远程部署服务器 方案

> 硬件：i5 12 代 CPU / 32 GB 内存 / 1 TB 硬盘
> 系统：保留 Windows + Docker Desktop
> 目标：本机运行 Jenkins，并对外提供"远程部署区"，让其他电脑通过 Jenkins 流水线把 Docker 应用部署到这台机器上。

---

## 一、整体架构

```
┌──────────────────────────────────────────────────────────────┐
│              Windows 11/10 (宿主机, 静态局域网 IP)           │
│                                                              │
│  ┌──────────────────────┐    ┌────────────────────────────┐  │
│  │ Docker Desktop       │    │ 远程部署区 (Deploy Zone)   │  │
│  │ (WSL2 后端)          │    │ 业务容器 (由 Jenkins 推送) │  │
│  │                      │    │                            │  │
│  │  jenkins (容器)      │    │  app-a, app-b, app-c ...   │  │
│  │   ├─ 8080 Web UI     │    │  数据库/中间件容器         │  │
│  │   └─ 50000 Agent     │    │                            │  │
│  │                      │    │  统一接入: Nginx/Traefik   │  │
│  └──────────┬───────────┘    └────────────┬───────────────┘  │
│             │                             │                  │
│             └────────── docker.sock ──────┘                  │
│                  (Jenkins 通过它把容器跑在宿主 Docker 上)    │
└──────────────────────────────────────────────────────────────┘
              ▲                                ▲
              │ 8080/HTTPS                     │ 80/443/业务端口
              │                                │
        其他开发电脑 ─────── 局域网 ───────  最终用户
```

**核心思路**

- Jenkins 本身跑在 Docker 容器里，方便迁移、备份和升级。
- Jenkins 容器**挂载 `docker.sock`**，让它能直接调用宿主机的 Docker 引擎，把构建出来的镜像作为容器跑在同一台机器上 —— 这就是"远程部署区"。
- 所有业务容器加入一个固定的 Docker 网络 `deploy-net`，由前置反向代理（Nginx 或 Traefik）按域名/端口分发，避免端口冲突。
- 数据持久化通过 **命名卷 (named volume)** 或 **绑定挂载到 C:\srv 专用目录**，便于备份。

## 二、资源分配（推荐方案）

针对 i5-12 代 (一般 6P+4E 或 10 核混合) + 32 GB + 1 TB，结合"Jenkins 轻量 + 部署区为主"的诉求：

| 用途                     | CPU 限制     | 内存限制      | 磁盘                      |
| ------------------------ | ------------ | ------------- | ------------------------- |
| Windows + Docker Desktop 自身 | -            | 预留 ~6 GB    | 系统盘 ~100 GB            |
| Jenkins (controller)     | 不限/上限 4 核 | **4 GB**      | 卷 `jenkins_home` ~50 GB  |
| Jenkins 构建 Agent (可选) | 不限         | **4 GB**      | 临时工作目录 ~50 GB       |
| 远程部署区（业务容器总和） | 不限         | **~16-18 GB** | 数据目录 ~500 GB          |
| 反向代理 / 监控          | 1 核         | 1 GB          | -                         |
| 预留缓冲                 | -            | ~3 GB         | ~300 GB (镜像/日志/备份)  |

> WSL2 默认会吃掉所有内存，**必须**通过 `.wslconfig` 显式限制（见第三节）。

**Docker Desktop 资源建议**：分配给 WSL2 后端 **24 GB 内存、8 核 CPU**，Windows 自身留 ~8 GB。

## 三、环境准备

### 3.1 Windows 侧

1. 设置 Windows **电源选项 → 高性能**，关闭睡眠（`控制面板 → 电源选项 → 更改计算机睡眠时间 → 从不`）。
2. 给本机配 **静态局域网 IP**（路由器里做 MAC 地址绑定，或在网络适配器里手动设置）。假设为 `192.168.1.100`。
3. **关闭 Windows 自动更新重启**（组策略：`计算机配置 → 管理模板 → Windows 组件 → Windows 更新 → 配置自动更新` 设为"通知下载和安装"）。
4. 安装 [WSL2](https://learn.microsoft.com/zh-cn/windows/wsl/install)：
   ```powershell
   wsl --install
   wsl --set-default-version 2
   ```
5. 安装 [Docker Desktop for Windows](https://www.docker.com/products/docker-desktop/)，安装时勾选 **Use WSL 2 instead of Hyper-V**。

### 3.2 限制 WSL2 资源

在 `C:\Users\<你的用户名>\.wslconfig` 写入：

```ini
[wsl2]
memory=24GB
processors=8
swap=8GB
localhostForwarding=true
```

修改后在 PowerShell 执行 `wsl --shutdown`，再启动 Docker Desktop 生效。

### 3.3 Docker Desktop 设置

- **Settings → Resources → WSL Integration**：开启。
- **Settings → General**：
  - 勾选 `Start Docker Desktop when you sign in`
  - 勾选 `Open Docker Desktop when you sign in`（开机自启）
- **Settings → Docker Engine**，加入国内镜像加速（按需）：
  ```json
  {
    "registry-mirrors": [
      "https://docker.mirrors.ustc.edu.cn",
      "https://hub-mirror.c.163.com"
    ],
    "log-driver": "json-file",
    "log-opts": { "max-size": "50m", "max-file": "5" },
    "default-address-pools": [
      { "base": "172.30.0.0/16", "size": 24 }
    ]
  }
  ```
- **Settings → Software updates**：关闭自动更新，避免 Jenkins 工作中被打断。

### 3.4 准备目录结构

在 C 盘规划专用目录：

```
C:\srv\
├── jenkins\          # Jenkins home 持久化
├── deploy\           # 业务容器数据卷父目录
│   ├── app-a\
│   ├── app-b\
│   └── postgres\
├── proxy\            # Nginx/Traefik 配置
│   └── conf\
├── backups\          # 备份输出
└── compose\          # docker-compose 文件集中存放
    ├── jenkins\
    └── proxy\
```

PowerShell 一次性创建：

```powershell
$base = "C:\srv"
"jenkins","deploy","deploy\app-a","deploy\app-b","deploy\postgres","proxy\conf","backups","compose\jenkins","compose\proxy" | ForEach-Object {
  New-Item -ItemType Directory -Force -Path (Join-Path $base $_) | Out-Null
}
```

## 四、部署 Jenkins

### 4.1 创建专用 Docker 网络

```powershell
docker network create deploy-net
```

所有 Jenkins、反向代理、业务容器都加入这张网，互相用容器名直连。

### 4.2 Jenkins compose 文件

文件 `C:\srv\compose\jenkins\docker-compose.yml`：

```yaml
services:
  jenkins:
    # 使用官方 LTS 镜像；如要自带 docker CLI，可换 jenkins/jenkins:lts-jdk17 自行扩展
    image: jenkins/jenkins:lts-jdk17
    container_name: jenkins
    restart: unless-stopped
    user: root                       # 为了能访问挂进来的 docker.sock；生产可改用 docker 组方案
    ports:
      - "8080:8080"                  # Web UI
      - "50000:50000"                # 给外部 Agent 用
    environment:
      - TZ=Asia/Shanghai
      - JAVA_OPTS=-Xmx3g -Dhudson.model.DirectoryBrowserSupport.CSP= -Duser.timezone=Asia/Shanghai
    volumes:
      - jenkins_home:/var/jenkins_home
      # 关键：让 Jenkins 能调用宿主 Docker
      - /var/run/docker.sock:/var/run/docker.sock
      # 让 Jenkins 容器内可以直接用 docker 命令（从宿主复制二进制更省事，也可以在镜像里装）
    networks:
      - deploy-net
    deploy:
      resources:
        limits:
          memory: 4g

volumes:
  jenkins_home:
    driver: local
    driver_opts:
      type: none
      device: C:\srv\jenkins
      o: bind

networks:
  deploy-net:
    external: true
```

启动：

```powershell
docker compose -f C:\srv\compose\jenkins\docker-compose.yml up -d
```

### 4.3 在 Jenkins 容器内安装 docker CLI

`jenkins/jenkins:lts-jdk17` 镜像里默认没有 `docker` 命令。两种做法：

**做法 A（推荐，简单）**：使用社区镜像 [`jenkins/jenkins:lts-jdk17` + Dockerfile 扩展]：

新建 `C:\srv\compose\jenkins\Dockerfile`：

```dockerfile
FROM jenkins/jenkins:lts-jdk17
USER root
RUN apt-get update && \
    apt-get install -y --no-install-recommends ca-certificates curl gnupg lsb-release && \
    install -m 0755 -d /etc/apt/keyrings && \
    curl -fsSL https://download.docker.com/linux/debian/gpg | gpg --dearmor -o /etc/apt/keyrings/docker.gpg && \
    echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/debian $(lsb_release -cs) stable" \
       > /etc/apt/sources.list.d/docker.list && \
    apt-get update && \
    apt-get install -y --no-install-recommends docker-ce-cli docker-compose-plugin && \
    rm -rf /var/lib/apt/lists/*
USER jenkins
```

把 compose 里 `image:` 改为 `build: .`，重新 `docker compose up -d --build`。

**做法 B**：使用现成的 `jenkins/jenkins` + 在每条流水线里用 `docker` agent 跑构建（不在 Jenkins 容器里直接调 docker）。适合更"干净"的模式，但前期复杂。

### 4.4 首次初始化

1. 浏览器访问 `http://192.168.1.100:8080`。
2. 取出初始密码：
   ```powershell
   docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword
   ```
3. 选择 **Install suggested plugins**。
4. 创建管理员账号。
5. 系统配置 → URL 设置为 `http://192.168.1.100:8080/`。

### 4.5 必装插件

- **Pipeline** / **Pipeline: Stage View**
- **Git** / **GitHub** / **Gitea** / **GitLab**（按你用的代码托管选）
- **Docker Pipeline**
- **Credentials Binding**
- **Blue Ocean**（可选，更好看的 UI）
- **SSH Agent**（如果还需要 SSH 到其他机器部署）
- **Generic Webhook Trigger**（接外部 webhook 触发）

### 4.6 配置凭据

`Manage Jenkins → Credentials → System → Global` 中至少添加：

- Git 仓库账号 / 部署 Token
- Docker Registry 账号（如 Harbor/Docker Hub/阿里云 ACR）

## 五、远程部署区设计

### 5.1 反向代理（统一入口）

文件 `C:\srv\compose\proxy\docker-compose.yml`，使用 Traefik 自动发现容器：

```yaml
services:
  traefik:
    image: traefik:v3.1
    container_name: traefik
    restart: unless-stopped
    command:
      - --api.dashboard=true
      - --providers.docker=true
      - --providers.docker.network=deploy-net
      - --providers.docker.exposedbydefault=false
      - --entrypoints.web.address=:80
      - --entrypoints.websecure.address=:443
    ports:
      - "80:80"
      - "443:443"
      - "8081:8080"   # Traefik dashboard (仅内网开)
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - C:\srv\proxy\conf:/etc/traefik/dynamic
    networks:
      - deploy-net

networks:
  deploy-net:
    external: true
```

启动：

```powershell
docker compose -f C:\srv\compose\proxy\docker-compose.yml up -d
```

### 5.2 业务容器命名与暴露规约

- **网络**：所有由 Jenkins 部署的业务容器必须 `--network deploy-net`。
- **命名**：`<项目名>-<环境>`，例如 `app-a-dev`、`app-a-prod`，方便流水线幂等替换。
- **数据**：宿主机数据目录固定在 `C:\srv\deploy\<项目名>\`。
- **路由**：通过 Traefik 标签声明域名，例如：

  ```bash
  docker run -d --name app-a-dev \
    --network deploy-net \
    --restart unless-stopped \
    -v C:\srv\deploy\app-a:/data \
    --label traefik.enable=true \
    --label traefik.http.routers.app-a-dev.rule=Host(`app-a-dev.lan`) \
    --label traefik.http.services.app-a-dev.loadbalancer.server.port=8080 \
    yourrepo/app-a:${BUILD_NUMBER}
  ```

  开发同事在 hosts 里加一行 `192.168.1.100 app-a-dev.lan` 就能访问。

### 5.3 Jenkins 流水线模板（Docker 部署）

`Jenkinsfile` 示例，放进每个业务仓库：

```groovy
pipeline {
  agent any

  environment {
    APP_NAME   = 'app-a'
    ENV_NAME   = 'dev'
    IMAGE      = "registry.local/${APP_NAME}:${BUILD_NUMBER}"
    DATA_DIR   = "C:\\srv\\deploy\\${APP_NAME}"
    NETWORK    = 'deploy-net'
    HOST_RULE  = "Host(`${APP_NAME}-${ENV_NAME}.lan`)"
  }

  stages {
    stage('Checkout') {
      steps { checkout scm }
    }

    stage('Build Image') {
      steps {
        sh 'docker build -t $IMAGE .'
      }
    }

    stage('Push (可选)') {
      when { expression { return env.PUSH_REGISTRY == 'true' } }
      steps {
        withCredentials([usernamePassword(credentialsId: 'registry-cred',
                         usernameVariable: 'U', passwordVariable: 'P')]) {
          sh 'echo $P | docker login registry.local -u $U --password-stdin'
          sh 'docker push $IMAGE'
        }
      }
    }

    stage('Deploy') {
      steps {
        sh '''
          NAME=${APP_NAME}-${ENV_NAME}
          docker rm -f $NAME 2>/dev/null || true
          docker run -d --name $NAME \
            --network ${NETWORK} \
            --restart unless-stopped \
            -v "${DATA_DIR}":/data \
            --label traefik.enable=true \
            --label "traefik.http.routers.${NAME}.rule=${HOST_RULE}" \
            --label "traefik.http.services.${NAME}.loadbalancer.server.port=8080" \
            ${IMAGE}
        '''
      }
    }

    stage('Health Check') {
      steps {
        sh 'sleep 5 && docker ps --filter "name=${APP_NAME}-${ENV_NAME}" --format "{{.Status}}"'
      }
    }
  }

  post {
    failure {
      sh 'docker logs --tail=200 ${APP_NAME}-${ENV_NAME} || true'
    }
  }
}
```

> Windows 上 Jenkins 容器跑的是 Linux 镜像，所以流水线里使用 `sh` 而不是 `bat`。
> `C:\srv\deploy\...` 这类宿主路径会通过 Docker Desktop 自动转换为 WSL 内可识别的挂载，**前提是该盘符已在 Docker Desktop → Resources → File sharing 启用**。

### 5.4 让"其他电脑"通过 Jenkins 部署

有两种工作模式：

**模式 1：开发者只提交代码，由 Jenkins 自动构建 + 部署（推荐）**

1. 在 Gitea/GitLab/GitHub 仓库配置 webhook → `http://192.168.1.100:8080/github-webhook/`（或 generic webhook）。
2. Jenkins Job 设置 "GitHub hook trigger" 或 "Poll SCM"。
3. 开发者只需要 `git push`，剩下交给 Jenkins。

**模式 2：开发者手动触发 Jenkins**

1. Jenkins Job 设置 "This project is parameterized"，加 `BRANCH`、`ENV_NAME` 等参数。
2. 开发者打开 `http://192.168.1.100:8080/job/xxx/build` 选参数构建。
3. 也可以用 Jenkins CLI / API：
   ```powershell
   curl -X POST "http://192.168.1.100:8080/job/app-a/buildWithParameters?BRANCH=main" `
        --user "user:api_token"
   ```

**不推荐**：让其他开发者直接 `docker -H tcp://...` 远程操作宿主 Docker —— 风险大、难以审计。统一走 Jenkins 流水线才是正解。

## 六、网络访问方案

> 你提到"还不确定"，这里给出**先内网 → 后扩展公网**的渐进路线。

### 6.1 第一阶段：仅局域网

无须额外配置，开发者在同网段访问 `http://192.168.1.100:8080`、`http://app-a-dev.lan` 即可。

**Windows 防火墙放行**（PowerShell 管理员）：

```powershell
New-NetFirewallRule -DisplayName "Jenkins"  -Direction Inbound -LocalPort 8080 -Protocol TCP -Action Allow
New-NetFirewallRule -DisplayName "HTTP"     -Direction Inbound -LocalPort 80   -Protocol TCP -Action Allow
New-NetFirewallRule -DisplayName "HTTPS"    -Direction Inbound -LocalPort 443  -Protocol TCP -Action Allow
New-NetFirewallRule -DisplayName "Jenkins-Agent" -Direction Inbound -LocalPort 50000 -Protocol TCP -Action Allow
```

### 6.2 第二阶段：跨网段 / 居家办公 — 推荐 Tailscale

无须公网 IP，几条命令就能让外网电脑安全访问：

1. 在这台 Windows 上安装 [Tailscale](https://tailscale.com/download/windows)，登录账号。
2. 其他电脑也安装 Tailscale，登录同一账号。
3. 直接用这台机器的 Tailscale IP（如 `100.x.x.x`）访问 Jenkins，无需开端口、零配置。
4. 可启用 **MagicDNS** 用机器名访问。

替代品：**ZeroTier**、**frp（自建中转）**。

### 6.3 第三阶段：真公网暴露

只有确实需要时再做：

1. 路由器做端口映射（仅 80/443，**不要**直接暴露 8080）。
2. 域名解析到家庭/公司公网 IP（动态 IP 配 DDNS：阿里云 / Cloudflare API）。
3. Traefik 配置 Let's Encrypt 自动证书：
   ```yaml
   command:
     - --certificatesresolvers.le.acme.email=you@example.com
     - --certificatesresolvers.le.acme.storage=/letsencrypt/acme.json
     - --certificatesresolvers.le.acme.httpchallenge=true
     - --certificatesresolvers.le.acme.httpchallenge.entrypoint=web
   ```
4. Jenkins 走 HTTPS 反向代理，并在系统设置中启用 **Enable security → Matrix-based / RBAC**，强制登录。
5. 启用 fail2ban 思路（用 Traefik 的 plugin / Crowdsec）防爆破。

## 七、安全清单（必做）

- [ ] Jenkins 第一时间创建管理员账号，禁用匿名读 (`Configure Global Security`)。
- [ ] **不要**把 Jenkins 8080 直接暴露到公网；如需要，前面套 HTTPS 反向代理 + IP 白名单。
- [ ] Jenkins 凭据全部走 Credentials 系统，**禁止**把密码写进 Jenkinsfile。
- [ ] Docker socket 挂进 Jenkins 等于 root 权限，确保 Jenkins 只允许内部信任的人登录。
- [ ] 所有业务容器加 `--restart unless-stopped`，并设置 `--memory` `--cpus` 限额，防止某个跑挂拖死宿主。
- [ ] 数据卷只放在 `C:\srv\` 下，定期备份（见第八节）。
- [ ] Windows 启用 BitLocker 给 C 盘加密（防丢盘）。
- [ ] 给 Windows 账户设强密码，开启锁屏，**禁用自动登录**。
- [ ] 关闭不必要的 Windows 端口（445/3389 默认不要对外）。

## 八、备份与运维

### 8.1 备份脚本

`C:\srv\backups\backup.ps1`：

```powershell
$ErrorActionPreference = "Stop"
$ts = Get-Date -Format "yyyyMMdd-HHmmss"
$dst = "C:\srv\backups\$ts"
New-Item -ItemType Directory -Force -Path $dst | Out-Null

# 1. 备份 Jenkins (打包，避免文件占用就先停容器；轻度备份可不停)
docker run --rm -v jenkins_home:/data -v "${dst}:/backup" alpine `
  sh -c "tar czf /backup/jenkins_home.tgz -C /data ."

# 2. 备份业务数据目录
Compress-Archive -Path C:\srv\deploy\* -DestinationPath "$dst\deploy.zip" -Force

# 3. 备份 compose 配置
Compress-Archive -Path C:\srv\compose\*,C:\srv\proxy\* -DestinationPath "$dst\config.zip" -Force

# 4. 仅保留最近 14 份
Get-ChildItem C:\srv\backups -Directory |
  Sort-Object Name -Descending | Select-Object -Skip 14 |
  Remove-Item -Recurse -Force
```

用任务计划程序设为每天凌晨 3 点执行：

```powershell
$action  = New-ScheduledTaskAction -Execute "powershell.exe" -Argument "-NoProfile -File C:\srv\backups\backup.ps1"
$trigger = New-ScheduledTaskTrigger -Daily -At 3am
Register-ScheduledTask -TaskName "SrvBackup" -Action $action -Trigger $trigger -RunLevel Highest
```

### 8.2 监控（轻量推荐）

可选加上一组监控容器（约 1-2 GB 内存）：

- **Portainer CE**：可视化管理 Docker（`9000` 端口）
- **Dozzle**：实时看所有容器日志
- **Uptime Kuma**：服务可用性监控

最小化方式直接 `docker run` 起来，加入 `deploy-net` 即可。

### 8.3 日常排错清单

| 现象                              | 排查方向                                                              |
| --------------------------------- | --------------------------------------------------------------------- |
| Jenkins 跑流水线时 `docker: not found` | 4.3 节构建带 docker CLI 的镜像，或确认 `docker.sock` 挂载成功         |
| 部署的容器外网访问不到             | 防火墙放行 / Traefik 标签写错 / 容器没加入 `deploy-net`               |
| 宿主机内存爆掉                     | 检查 `.wslconfig` 是否生效；给业务容器加 `--memory` 限制              |
| Jenkins 升级后插件报错             | 用 6.1 备份恢复 `jenkins_home.tgz`，再回滚镜像 tag                    |
| 镜像越积越多吃硬盘                 | 定期 `docker image prune -af --filter "until=240h"`                   |
| WSL 占盘很大                       | `wsl --shutdown` 后 `Optimize-VHD` 压缩 `ext4.vhdx`                   |

## 九、实施顺序（Checklist）

按序完成，每步可单独验证：

1. [ ] Windows 改静态 IP、禁睡眠、禁自动重启

## 十、GitHub Webhook 配置（自动构建 + 部署）

> 目标：开发者 `git push` 到 GitHub → GitHub 通知 Jenkins → Jenkins 拉代码、构建镜像、替换部署区容器。

### 10.1 前置条件

- Jenkins 必须能被 GitHub 访问到。两种情形：
  - **A. Jenkins 已经有公网入口**（域名 / 公网 IP / Tailscale Funnel）：GitHub 直接 POST。
  - **B. Jenkins 只在内网**（你当前情况）：用 **smee.io** 或 **GitHub 自托管 webhook 转发**，下面 10.6 给方案。
- Jenkins 已装插件：**GitHub**、**GitHub Branch Source**、**Pipeline**、**Docker Pipeline**、**Credentials Binding**。

### 10.2 在 GitHub 生成 Personal Access Token (PAT)

供 Jenkins 拉代码 + 回写 commit status 用：

1. GitHub 右上头像 → **Settings → Developer settings → Personal access tokens → Tokens (classic)** → **Generate new token (classic)**。
2. 勾选权限：
   - `repo`（完整仓库访问，私有仓必需）
   - `admin:repo_hook`（让 Jenkins 自动管理 webhook，可选但推荐）
   - `read:org`（如果代码在组织下）
3. 生成后复制 token（只显示一次）。

> 也可以用更安全的 **Fine-grained token**：仅授权目标仓库 + `Contents: Read`、`Metadata: Read`、`Webhooks: Read & write`、`Commit statuses: Read & write`。

### 10.3 在 Jenkins 添加 GitHub 凭据

`Manage Jenkins → Credentials → System → Global` → **Add Credentials**：

| 字段        | 值                                              |
| ----------- | ----------------------------------------------- |
| Kind        | `Username with password`                        |
| Username    | 你的 GitHub 用户名                               |
| Password    | 上一步生成的 PAT                                 |
| ID          | `github-pat` （Jenkinsfile 里会用到这个 ID）    |
| Description | `GitHub PAT for Jenkins`                        |

再加一份 **Secret text** 类型：

| 字段        | 值                                       |
| ----------- | ---------------------------------------- |
| Kind        | `Secret text`                            |
| Secret      | 同一个 PAT                                |
| ID          | `github-token`                           |

> 两份是因为不同插件用的凭据类型不同：Git checkout 要 `username/password`，GitHub 插件本身要 `secret text`。

### 10.4 配置 Jenkins 全局 GitHub 服务器

`Manage Jenkins → System` → 找到 **GitHub** 区域 → **Add GitHub Server**：

- Name: `github`
- API URL: `https://api.github.com`
- Credentials: 选 `github-token`
- 勾选 **Manage hooks**（让 Jenkins 自动给仓库加/删 webhook）
- 点 **Test connection**，应显示 `Credentials verified for user xxx, rate limit: 5000`

### 10.5 创建 Multibranch Pipeline 任务（推荐）

比单分支 Pipeline 更适合 GitHub 工作流，能自动发现分支和 PR。

1. Jenkins 首页 → **New Item** → 名字 `app-a` → 选 **Multibranch Pipeline** → OK。
2. **Branch Sources** → **Add source → GitHub**：
   - Credentials: `github-pat`
   - Repository HTTPS URL: `https://github.com/<your-org>/<repo>.git`
   - **Behaviors** 保留默认（Discover branches、Discover PRs）。
3. **Build Configuration**：
   - Mode: `by Jenkinsfile`
   - Script Path: `Jenkinsfile`
4. **Scan Repository Triggers**：勾选 `Periodically if not otherwise run`，设 1 day（兜底，避免 webhook 偶发丢失）。
5. 保存。Jenkins 会自动：
   - 扫描所有分支
   - **在 GitHub 仓库里创建 webhook**（指向 `http(s)://<jenkins-url>/github-webhook/`）
   - 立即跑一次构建

如果 Jenkins 自动建 webhook 失败（通常是因为 GitHub 访问不到 Jenkins），手动建即见 10.6。

### 10.6 内网 Jenkins 接收 GitHub Webhook 的三种做法

#### 做法 1：smee.io（最简单，零成本）

GitHub 官方推荐的开发用转发器，把公网 webhook 转发到本地 Jenkins。

1. 打开 https://smee.io/ → 点 **Start a new channel**，得到一个 URL，例如 `https://smee.io/AbCdEfGh1234`。
2. 在宿主机上跑 smee 客户端容器，把流量转发进 Jenkins：

   ```powershell
   docker run -d --name smee --restart unless-stopped `
     --network deploy-net `
     deltaprojects/smee-client `
     --url https://smee.io/AbCdEfGh1234 `
     --target http://jenkins:8080/github-webhook/
   ```

3. GitHub 仓库 → **Settings → Webhooks → Add webhook**：
   - Payload URL: `https://smee.io/AbCdEfGh1234`
   - Content type: `application/json`
   - Secret: 见 10.7
   - Which events: `Just the push event`（或 `Send me everything`）
   - Active: 勾上
4. 测试：`git push` 一下，在 Jenkins 任务页应能看到自动触发的构建。

> 缺点：依赖 smee.io 第三方服务，不适合生产；适合个人/小团队过渡期。

#### 做法 2：Tailscale Funnel（推荐，安全且免费）

如果第六节装了 Tailscale，可以一条命令把 Jenkins 暴露到公网（带 HTTPS 证书）：

```powershell
# 在宿主机 PowerShell 执行（需 Tailscale ACL 启用 funnel）
tailscale funnel --bg --https=443 http://localhost:8080
```

得到一个形如 `https://<机器名>.<tailnet>.ts.net/` 的公网 URL，把 GitHub webhook Payload URL 设为：

```
https://<机器名>.<tailnet>.ts.net/github-webhook/
```

> 末尾的 `/github-webhook/` **必须带斜杠**，这是 GitHub 插件固定的入口。

#### 做法 3：路由器端口映射 + DDNS

仅当你愿意把 Jenkins 真正暴露到公网时用，做完务必配 HTTPS（第六节 6.3）+ Jenkins 强密码 + 仅放行 GitHub 的 IP 段（见 10.8）。

### 10.7 配置 Webhook Secret（强烈建议）

防止伪造请求：

1. 生成随机串：
   ```powershell
   -join ((48..57) + (65..90) + (97..122) | Get-Random -Count 40 | % {[char]$_})
   ```
2. GitHub webhook 表单 **Secret** 填入这个串。
3. Jenkins → **Manage Jenkins → System → GitHub → Advanced → Shared secrets** → **Add → Secret text**，把这个串保存为 `Secret text` 凭据，并在下拉里选中。
4. 之后 Jenkins 会校验 GitHub 请求头里的 `X-Hub-Signature-256`，签名不对的请求直接丢弃。

### 10.8 GitHub IP 白名单（仅公网暴露时）

GitHub 的 webhook 来源 IP 段公开在 https://api.github.com/meta 的 `hooks` 字段里。
如果用 Traefik，可以加一条中间件只放行这些段：

```yaml
# C:\srv\proxy\conf\github-allowlist.yml
http:
  middlewares:
    github-only:
      ipWhiteList:
        sourceRange:
          - 192.30.252.0/22
          - 185.199.108.0/22
          - 140.82.112.0/20
          - 143.55.64.0/20
          # 以 https://api.github.com/meta 为准，定期更新
```

然后在 Jenkins 路由的 Traefik 标签里追加 `traefik.http.routers.jenkins.middlewares=github-only@file`。

### 10.9 Jenkinsfile 改造（带 GitHub Commit Status 回写）

在第五节的 Jenkinsfile 基础上加两点：使用 GitHub 凭据 checkout，并把构建结果写回 PR/commit。

```groovy
pipeline {
  agent any

  options {
    // 自动把构建状态回写到 GitHub commit
    githubProjectProperty(projectUrlStr: 'https://github.com/<your-org>/<repo>/')
    buildDiscarder(logRotator(numToKeepStr: '20'))
    timeout(time: 30, unit: 'MINUTES')
  }

  triggers {
    // Multibranch 不需要这行；普通 Pipeline 加上以接收 GitHub push
    githubPush()
  }

  environment {
    APP_NAME = 'app-a'
    ENV_NAME = "${env.BRANCH_NAME == 'main' ? 'prod' : 'dev'}"
    IMAGE    = "registry.local/${APP_NAME}:${env.BRANCH_NAME}-${env.BUILD_NUMBER}"
    DATA_DIR = "C:\\srv\\deploy\\${APP_NAME}"
    NETWORK  = 'deploy-net'
  }

  stages {
    stage('Checkout') {
      steps {
        checkout scmGit(
          branches: [[name: "*/${env.BRANCH_NAME}"]],
          userRemoteConfigs: [[
            url: 'https://github.com/<your-org>/<repo>.git',
            credentialsId: 'github-pat'
          ]]
        )
      }
    }

    stage('Build Image') {
      steps { sh 'docker build -t $IMAGE .' }
    }

    stage('Deploy') {
      // 只在 main / develop 自动部署，PR 仅做 build 验证
      when { anyOf { branch 'main'; branch 'develop' } }
      steps {
        sh '''
          NAME=${APP_NAME}-${ENV_NAME}
          docker rm -f $NAME 2>/dev/null || true
          docker run -d --name $NAME \
            --network ${NETWORK} \
            --restart unless-stopped \
            -v "${DATA_DIR}":/data \
            --label traefik.enable=true \
            --label "traefik.http.routers.${NAME}.rule=Host(\`${APP_NAME}-${ENV_NAME}.lan\`)" \
            --label "traefik.http.services.${NAME}.loadbalancer.server.port=8080" \
            ${IMAGE}
        '''
      }
    }
  }

  post {
    success { echo "OK: ${IMAGE}" }
    failure {
      sh 'docker logs --tail=200 ${APP_NAME}-${ENV_NAME} || true'
    }
  }
}
```

### 10.10 验证流程

1. GitHub 仓库 → **Settings → Webhooks**，点进 webhook，**Recent Deliveries** 应能看到一条 `200` 的请求；如果是红色，点进去看 Response，常见错误：
   - `403 No valid crumb` → Jenkins 安全配置问题，去 **Configure Global Security** 关掉 "Prevent Cross Site Request Forgery exploits" 对 `/github-webhook/` 的影响（默认已豁免）。
   - `404` → URL 末尾少了 `/`，应为 `.../github-webhook/`。
   - `Connection timeout` → GitHub 访问不到 Jenkins，回 10.6 选转发方案。
2. Jenkins 任务页应自动出现新构建。
3. 回到 GitHub 仓库 commit 列表，应看到每个 commit 旁有 ✅ 或 ❌ 状态图标，PR 页能看到 "Jenkins / app-a — Pipeline succeeded" 检查项。

### 10.11 常见进阶

- **只在改了某些路径时才构建**：Jenkinsfile `when { changeset "src/**" }`。
- **PR 评论触发重跑**：装插件 **GitHub PR Comment Build**，在 PR 里回 `rebuild` 即可重跑。
- **多仓库共用同一套流水线**：用 **Shared Library**（`@Library('your-lib') _`），把 Jenkinsfile 缩成几行。
- **构建产物回传 GitHub Release**：用 `gh release create` + PAT，在 `post { success { ... } }` 里发布。
